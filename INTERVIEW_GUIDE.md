## 1. PROJECT OVERVIEW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### What SecPay is
SecPay is a secure, cloud-ready, API-based Utility Billing Platform. It enables users to view, manage, track, and pay their utility bills seamlessly through a consolidated dashboard, facilitated by an integrated payment gateway.

### What problem it solves
Managing utility bills across multiple external providers often leads to fragmented tracking, missed deadlines, and disjointed payment flows. SecPay consolidates this process into a single platform. It handles the complete lifecycle from invoice generation and penalty calculation to secure payment settlement, preventing late fees and ensuring reliable record-keeping.

### Why it was built
SecPay was built to demonstrate full-stack engineering capabilities, incorporating complex business logic, third-party API integration, robust relational database design, and modern backend security best practices. It serves as a comprehensive system showcasing API-first architecture, complex state machines (payment lifecycles), and transactional data consistency.

### High-level architecture
The platform strictly follows a decoupled three-tier architecture:
- **Presentation Layer (Frontend):** Pure HTML, CSS, and Vanilla JavaScript. It communicates entirely via REST APIs, handling DOM manipulation and JWT storage.
- **Application Layer (Backend):** A Python-based FastAPI server serving RESTful endpoints, handling the business logic, JWT validation, and orchestrating external API combinations.
- **Data Layer (Database):** A relational MySQL database interacting with the application via SQLAlchemy ORM.
- **Third-Party Services (Payment Gateway):** Razorpay handles payment interface generation, checkout processing, and cryptographic signature validation.

## 2. TECH STACK
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- **Backend framework (FastAPI):** Chosen for its extreme high performance natively over ASGI, built-in async/await support, Pydantic type validation, and automatic OpenAPI generation.
- **Frontend stack (HTML, CSS, JS):** Vanilla frontend ensures a lightweight, lightning-fast client-side experience without modern framework overhead, focused purely on standard web APIs and asynchronous data fetching.
- **Database (MySQL):** A robust relational database ideal for ensuring strict ACID compliance, referential integrity, and safe transactional boundaries, which are crucial for a financial application.
- **ORM (SQLAlchemy):** Provides a Pythonic way to interact with the DB, perfectly abstracting SQL queries while managing database sessions, connection pooling, and strict schemas.
- **Payment gateway (Razorpay):** An industry-leading payment processor utilized for secure transactions, offering embedded checkout UI elements and cryptographic verification layers.
- **Authentication method (JWT):** JSON Web Tokens offer a robust, stateless authentication mechanism securing REST API endpoint boundaries without generating memory bloat via server-side session storage.
- **Password hashing method (bcrypt):** A highly secure, computationally expensive hashing algorithm that actively guards user passwords against brute-force, rainbow table, and timing attacks.
- **API documentation (OpenAPI / Swagger):** Automatically generated by FastAPI via Python typings, equipping frontend developers with self-updating, interactive endpoint testing documentation.

## 3. FOLDER STRUCTURE EXPLANATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### Backend
- **`backend/app/models/`**
  - *Contains:* SQLAlchemy database models (Python classes).
  - *Why it exists:* Translates python runtime objects (User, Bill) directly into structured SQL tables. Defines schemas, column constraints, and relational Foreign Keys.
- **`backend/app/routes/`**
  - *Contains:* FastAPI router definitions (the endpoint controllers).
  - *Why it exists:* Maps raw HTTP requests (GET, POST) to specific functions. Manages ingress payload parsing and egress response formatting, keeping transport separate from logic.
- **`backend/app/services/`**
  - *Contains:* Core business logic and database interactions.
  - *Why it exists:* This abstracts operations like calculating penalties or verifying payments, keeping route handlers extremely thin to adhere to single-responsibility definitions.
- **`backend/app/schemas/`**
  - *Contains:* Pydantic models.
  - *Why it exists:* For granular data validation, serialization, and deserialization. It serves as a strict firewall for ensuring request body formats are perfectly accurate and masks private data in responses.
- **`database.py`**
  - *Contains:* The database connection set-up, SQLAlchemy engine, and session dependency generator.
  - *Why it exists:* Standardizes connection properties to the MySQL server and dictates session lifecycles for FastAPI path operations.
- **`main.py`**
  - *Contains:* The root FastAPI application wrapper, CORS definitions, and router aggregation.
  - *Why it exists:* The definitive entry point of the server that bootstraps the entire application framework.
- **`settings.py`** (or `config`)
  - *Contains:* Environment variables and configuration structures.
  - *Why it exists:* Ingests critical, sensitive credentials (DB URIs, Razorpay keys, JWT secrets) from local `.env` files safely.

### Frontend
- **Frontend Assets (HTML, CSS, Scripts):**
  - *Contains:* Splitting visuals from logic. The `scripts` directory holds logic tied strictly to specific page actions (e.g. `bills.js`, `dashboard.js`).
  - *How it connects:* Binds event listeners to UI buttons, dynamically updates the DOM, injects `Bearer <token>` headers into `fetch()` requests targeting the backend, and invokes the Razorpay frontend script.

### Where Razorpay Logic Lives
- *Frontend:* Uses the Razorpay Checkout script mapped inside checkout javascript functions to launch the UI widget. 
- *Backend:* Lives strictly inside `payment_service` (for verifying cryptography) and dedicated `payment_routes`.

## 4. DATABASE DESIGN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### `users` table
- *Columns:* `id` (PK), `email` (Unique), `hashed_password`, `is_active`, `created_at`.
- *Purpose:* Contains all registered actors mapped to their credentials.

### `providers` table
- *Columns:* `id` (PK), `name` (Unique), `service_type` (Electricity, Water, internet).
- *Purpose:* The central catalog of utility companies.

### `bills` table
- *Columns:* `id` (PK), `user_id` (FK), `provider_id` (FK), `amount`, `due_date`, `status` (PENDING, OVERDUE, PAID), `penalty_amount`.
- *Relationships:* Belongs to `users` AND belongs to `providers`.
- *Transitions:* Moves dynamically from `PENDING` -> `OVERDUE` (based on chronometry), and `PENDING/OVERDUE` -> `PAID` (triggered solely by transactional payment verification).

### `payments` table
- *Columns:* `id` (PK), `bill_id` (FK), `razorpay_order_id`, `razorpay_payment_id`, `status` (SUCCESS, FAILED), `amount_paid`, `paid_at`.
- *Relationships:* Belongs directly strictly to `bills` (One-to-Many).
- *Transitions:* Created via Razorpay workflow boundaries. A successful payment insert locks the parent Bill's status to `PAID`.

## 5. AUTHENTICATION FLOW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. **Register:** User issues email/password. System hashes it via `bcrypt` instantly and stores the derived string.
2. **Login:** User issues credentials. Backend pulls the hash, compares iterative signatures, and confirms identity.
3. **JWT Creation:** System mints a JWT containing `sub: user_id` and an expiration timestamp `exp`, cryptographically signed by an internal secret.
4. **Token Storage & Usage:** Frontend hoards this token inside `localStorage`/`sessionStorage` and injects it into every API `Authorization` header.
5. **Protected Routes:** Endpoints demanding authorization wrap behind a `Depends(get_current_user)` pipeline interceptor. This validates the signature and asserts the token hasn't expired before executing business logic.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
## 6. BILL LIFECYCLE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- **PENDING:** The initial resting state when invoices are generated. Expected payment before `due_date`.
- **OVERDUE:** Occurs fundamentally when `current_date > due_date`. This dynamically applies financial penalties.
- **PAID:** Final state. Outstanding balance is 0. 
- **Penalty Calculation:** If overdue, a strictly defined business rule applies a percentage or static fee structure appended dynamically to required totals inside retrieval functions or UI endpoints.
- **Status Trigger:** The `PAID` state is impossible to trigger manually. It transforms ONLY upon an atomic, successful database insert of a verified Razorpay payment receipt logging a total matches.

## 7. PAYMENT FLOW (RAZORPAY)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. **Create order:** User clicks "Pay". Backend requests a secured `order_id` from Razorpay matching the exact outstanding amounts on a specific `bill_id`.
2. **Open checkout:** Frontend utilizes that `order_id` alongside the platform public Razorpay Key to initialize the UI sequence.
3. **Handle success:** User fills payment logic > UI intercepts success callback directly. Returns triplet payload: `order_id`, `payment_id`, `signature`.
4. **Verify signature:** Triplet is strictly POSTed back to FastAPI.
5. **Insert SUCCESS payment:** Backend applies HMAC-SHA256 calculations independently combining `order_id` & `payment_id` merged with the hidden `API_SECRET`. If internal calculation == user-provided `signature`, validation passes.
6. **Update bill:** Inside a locked database transaction, a success log is written, and the target Bill updates to `PAID`.
7. **Handle FAILED case:** Declinations fire a failure callback capturing the state logic independently without modifying Bill data.
8. **Dashboard metrics update:** Fetch loops re-run, migrating frontend numerical elements correctly.

**Key Explanations for Interviewers:**
- *Why signature verification is important:* It prevents frontend manipulators (attackers using Postman) from triggering "payment completed" cascades without handing money to Razorpay.
- *Why secret key is hidden:* To prevent attackers from generating perfectly valid "fraudulent" cryptography that bypasses validation. 
- *Why verification on backend:* Backend processes live in protected memory enclaves off-limits to external network inspection.

## 8. FAILED PAYMENT HANDLING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- **When FAILED is logged:** Whenever users abandon pop-ups intuitively, or bank transactions suffer immediate decline codes. Handled properly so logs track attempted interactions.
- **Validation failures:** If a bad actor tampers with signatures, the transaction rollback completely triggers an HTTP 400. Payments are utterly discarded internally.
- **Duplicate Prevention:** Before verification logic finalizes, DB searches using `razorpay_order_id` occur. If it already equates to `SUCCESS`, idempotency rejects the request entirely.

## 9. DASHBOARD METRICS LOGIC
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- **Outstanding:** Reconstructed by the equation `SUM(amount + dynamically_applied_penalty)` where DB `status` logic states are distinctly NOT `PAID`.
- **Paid Count:** Derived via raw `COUNT()` aggregations against any entity distinctly attached to the logged user enforcing SQL `WHERE status = 'PAID'`.
- **Failed Count:** Formulated tracking across the `payments` join, counting exclusively failed attempts. 
- **Overdue Count:** Tracks derived chronological conditions where `due_date < todays_date` alongside NOT `PAID`.

## 10. SECURITY CONSIDERATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- **Password hashing:** Never store plain texts. Utilize CPU-Intensive iterating logic like `bcrypt` so compromised DB lists are useless against reverse engineering.
- **JWT usage:** Circumvents heavy CSRF vulnerabilities bound to typical cookie infrastructures.
- **Environment variables:** Extracts deployment specifics limiting Github scrape-theft bot exploits targeting AWS/Razorpay constants.
- **Razorpay secret handling:** Never leaves the backend memory space.
- **Signature verification:** Installs a non-refutable mathematical "Proof-Of-Concept". 
- **Data masking in UI:** Pydantic safely eliminates `hashed_password` arrays from egress paths seamlessly. User objects stay lightweight.

## 12. INTERVIEW Q&A PREPARATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

**Q1: Why rely on FastAPI rather than mature ecosystems like Django or Express.js?**
*Answer:* Performance and modernization. FastAPI leverages ASGI for extremely rapid asynchronous execution speeds compared to older WSGI architectures. Its Pydantic auto-validation limits boilerplates, whilst its out-of-the-box Swagger/OpenAPI setup accelerated frontend integration.

**Q2: What happens if I manually modify the frontend codebase to skip the Razorpay `checkout` and send a fake "success" payload directly to your `/payment/verify` route?**
*Answer:* It would cleanly fail exactly at the cryptographic boundary. The verification endpoint takes the Razorpay IDs and hashes them uniquely utilizing the system's hidden `API_SECRET`. A fake payload lacks the mathematical integrity capable of mimicking HMAC-SHA256 signatures derived exclusively inside external Razorpay systems - therefore dropping a 400 Bad Request error immediately.

**Q3: Walk me through how you prevent database connection starvation during high API traffic?**
*Answer:* I rely directly on SQLAlchemy's connection pooling structures managed via FastAPI injection dependencies. Utilizing a `Depends(get_db)` generator guarantees active thread locking is exclusively attached per active HTTP lifecycle frame. Concurrency utilizes generators efficiently terminating connections securely within reliable `finally` closures or teardowns.

**Q4: How does SecPay ensure a bill doesn't get partially paid or updated if an error crashes the database inside the payment pipeline?**
*Answer:* Strict transactional atomicity. Inside the logic flow, inserting the Payment log alongside mutating the Bill status wrapper exists within a solitary DB Session commit framework. Any syntax failure, latency timeout, or explicit internal rollback drops the entire sequence block—ensuring we never encounter "Orphan Payments" nor "Paid bills lacking payment receipts". 

**Q5: Could you explain why your Database structures "Providers" away from "Bills" instead of flat columns?**
*Answer:* First Normal Form principles (1NF/3NF). If ten-thousand users utilize "Tata Power", keeping it statically flat inside Bills introduces massive redundancy wastes, prevents simple aggregation (e.g. reporting on Tata Power collectively), and risks UI typo fragmentation. A structured constraint preserves absolute data sanitation. 

**Q6: What is the most critical logic applied within your Bill states dynamically?**
*Answer:* Time-evaluations. The "Overdue" element fundamentally evaluates based identically against continuous runtime chronologies compared against database columns `due_dates` - avoiding heavy nightly "status update" CRON jobs natively, by dynamically appending states/penalties during retrieval phases based on current system UTC metrics.

**Q7: How hard is it to rotate out Razorpay if business demands a switch to Stripe?**
*Answer:* Relatively decoupled. Implementation logic abstracts heavily outside generic routes. Writing a "StripePaymentService" utilizing internal adapters and simply modifying frontend initialization UI structures limits breaking logic structures attached directly onto database mutations safely. 

**Q8: Explain how Idempotency actually guards against duplication in payment processing architectures?**
*Answer:* In SecPay, if frontend network lag causes accidental double-submission of an identical verified payload sequence - Idempotent engineering guarantees side-effect eliminations. We inspect `razorpay_order_id` existence aggressively. A matched hash collision aborts processing silently, avoiding billing a client twice mathematically. 

**Q9: If scaling, where is the most intensive system bottleneck?**
*Answer:* Dashboard metric aggregations. Utilizing aggregate SQL functions (`SUM()`, `COUNT()`) natively against heavy datasets lags execution parameters sequentially. Refactoring this into cached instances mapped locally over Redis instances matching invalidation upon state triggers guarantees latency structures dissolve globally. 

**Q10: Explain why JWTs specifically compliment secure microservice setups better than session IDs?**
*Answer:* Eliminating database trips. A Session ID strictly forces server-side memory querying sequentially per auth request to derive credentials. JWTs contain the claims payloads bundled self sufficiently; bounded strictly by signature verification natively scaling CPU validation architectures dynamically rather than generating DB memory friction pools natively.
